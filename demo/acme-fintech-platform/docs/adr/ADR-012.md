# ADR-012: Resilient authentication with OAuth fallback and refresh-token rotation

- **Status:** Accepted
- **Date:** 2025-03-22
- **Deciders:** Priya Nair (Staff Eng, Platform), Marcus Lee (Security), Dana Whitfield (SRE)
- **Related:** INC-284, PR #842

## Context

On 2025-03-18 our OAuth providers (Google and Azure) suffered a 17-minute partial
outage (see INC-284). Because `authenticateUser()` had a hard dependency on the
identity provider for every request, active enterprise users were force-logged-out
mid-session. Several customers escalated; one lost an in-progress payment flow.

We need authentication that survives transient identity-provider failures without
weakening security.

## Decision

1. **Cache sessions in Redis.** An already-authenticated user is validated against
   a cached session, so a provider blip does not invalidate an active session.
2. **Retry the provider with bounded backoff**, then **fall back to the cached
   session** rather than force a logout.
3. **Rotate refresh tokens** on every issue to close the replay-attack window that
   Security flagged during the post-mortem.

## Alternatives considered

- **Do nothing / hard dependency on the provider.** Rejected: this is exactly what
  caused the INC-284 outage impact.
- **Dramatically longer JWT TTLs.** Rejected: reduces outage exposure but widens the
  token-replay attack surface — the opposite of what Security wanted.
- **Multi-provider failover only.** Rejected as insufficient on its own: both
  providers degraded simultaneously during INC-284.

## Consequences

- Redis becomes load-bearing for authentication (and, later, payments idempotency).
- `authenticateUser()` is intentionally more complex; it must not be "simplified"
  back to a single OAuth call. This is enforced by tests in `auth.service.spec.ts`.
